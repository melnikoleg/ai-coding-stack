# Maintain ARCHITECTURE.md from the codebase-memory knowledge graph

You maintain a generated architecture document, `ARCHITECTURE.md`, at the
project root. Everything in it must come from the `codebase-memory` MCP
server's graph data (plus the project README for the overview) — never from
guesswork.

## Preflight (always)

1. Verify the `codebase-memory` MCP tools are available (e.g. `list_projects`).
   If they are not, STOP and tell the user to install the stack
   (https://github.com/melnikoleg/ai-coding-stack, `install-ai-coding-stack.sh`)
   or check their MCP config. Do NOT write the document without the graph.
2. Call `index_repository` on this project's root directory, then poll
   `index_status` until indexing completes.

## Mode selection

Use **full mode** if ANY of these hold:
- `$ARGUMENTS` contains `full`
- `ARCHITECTURE.md` does not exist
- its first line is not a valid marker of the form
  `<!-- arch-doc commit=<sha> state=<hash> -->`
- `git cat-file -e <marker-sha>^{commit}` fails (the base commit was
  rebased/squashed away)

Otherwise use **incremental mode** with `BASE = <marker commit>`.

## Full mode

Gather data in this order:

1. `get_architecture` — languages, packages, module clusters, HTTP routes.
   This feeds most sections.
2. Entry points: `search_graph` for Route nodes and for Function/Method nodes
   with names matching `main|cli|handler|serve|app|start|run`. If the result
   is thin, supplement with one `query_graph` query for public functions with
   zero incoming CALLS in the largest clusters. Keep at most 15 entries.
3. Module dependencies: `query_graph` with a Cypher query like
   `MATCH (a:Module)-[:IMPORTS]->(b:Module) RETURN a.name, b.name`. If the
   project has more than ~25 modules, aggregate edges to cluster level (one
   diagram node per cluster, edge labels = import counts).
4. Dead code: `search_graph` for Function/Method nodes with zero incoming
   CALLS edges. Exclude tests, exported/public API symbols, and anything
   already listed as an entry point. Keep at most 30 rows.

Then write `ARCHITECTURE.md` with exactly this structure:

1. Marker line (first line, see below).
2. `# Architecture` title and a note: generated by `/architecture` and kept
   fresh automatically — do not hand-edit, edits will be overwritten.
3. `## Overview` — 2–3 paragraphs on what the system is and how it is shaped,
   derived from cluster/package names and the project README.
4. `## Languages & Packages` — a table.
5. `## Module Map` — one subsection per cluster: its purpose and key
   files/modules.
6. `## Key Entry Points` — table: symbol, file, role.
7. `## HTTP Routes` — table: method, path, handler symbol, file. Omit the
   section entirely if there are no routes.
8. `## Module Dependency Diagram` — a mermaid `flowchart TD` block built from
   step 3 data (max ~25 nodes; collapse to clusters above that).
9. `## Dead Code Candidates` — table, explicitly labelled
   "candidates (heuristic — zero callers in the indexed graph)". Omit if none.

Finally, run `bash .claude/hooks/arch-doc-state.sh` (no arguments) and put its
output into the marker as the FIRST line of the file:

```
<!-- arch-doc commit=<sha> state=<hash> -->
```

## Incremental mode

1. Call `detect_changes` with `base = <marker commit>` to map the diff to
   affected symbols/files/modules.
2. If the affected set is empty (only non-source churn), just refresh the
   marker line (run `bash .claude/hooks/arch-doc-state.sh` with no arguments)
   and report "no architecture-relevant changes".
3. If the affected set is large (more than ~30% of modules or ~50 files), fall
   back to full mode.
4. Otherwise update only what changed:
   - Module Map subsections for affected clusters;
   - HTTP Routes only if route-defining files changed;
   - the dead-code query re-run scoped to affected modules;
   - the mermaid diagram only if imports changed (module added/removed,
     import edges changed);
   - Overview and Languages & Packages refreshed from one `get_architecture`
     call (it is cheap).
5. Refresh the marker line: run `bash .claude/hooks/arch-doc-state.sh` (no
   arguments) and replace the first line of `ARCHITECTURE.md` with its output.

If `detect_changes` errors or behaves unexpectedly, fall back to full mode.

## Rules

- Do not modify any files other than `ARCHITECTURE.md`.
- Keep the document factual and compact; no speculation about intent that the
  graph cannot support.
- Hard caps: 25 diagram nodes, 30 dead-code rows, 15 entry points.
