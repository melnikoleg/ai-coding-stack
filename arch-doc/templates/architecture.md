# Maintain ARCHITECTURE.md from the codebase-memory knowledge graph

You maintain a generated architecture document, `ARCHITECTURE.md`, at the
project root. Everything in it must come from the `codebase-memory` MCP
server's graph data (plus the project README for the overview) ‚Äî never from
guesswork.

## Preflight (always)

1. Verify the `codebase-memory` MCP tools are available. If they are not,
   STOP and tell the user to install the stack
   (https://github.com/melnikoleg/ai-coding-stack, `install-ai-coding-stack.sh`)
   or check their MCP config. Do NOT write the document without the graph.
2. Call `index_repository` with `repo_path` = absolute project root and
   `name` = the repo directory name. In current server versions the call is
   synchronous and returns `status: "indexed"`; if your server version also
   exposes `index_status`, poll it until indexing completes.
3. Every other tool requires a `project` parameter ‚Äî pass the project name
   returned by `index_repository`.

## Mode selection

Use **full mode** if ANY of these hold:
- `$ARGUMENTS` contains `full`
- `ARCHITECTURE.md` does not exist
- its first line is not a valid marker of the form
  `<!-- arch-doc commit=<sha> state=<hash> -->`
- `git cat-file -e <marker-sha>^{commit}` fails (the base commit was
  rebased/squashed away)

Otherwise use **incremental mode** with `BASE = <marker commit>`.

## Full mode

Gather data in this order:

1. `get_architecture` ‚Äî one call returns `languages`, `packages`,
   `entry_points`, `routes`, `hotspots`, `boundaries` (module‚Üímodule call
   counts), `layers`, `clusters`, and `file_tree`. This feeds most sections.
2. Entry points: start from `entry_points` in the `get_architecture` result.
   If thin, supplement with `search_graph` for Route nodes and for
   Function/Method nodes named `main|cli|handler|serve|app|start|run`. Keep
   at most 15 entries.
3. Module dependencies for the diagram: use `boundaries` from
   `get_architecture` (edges labelled with call counts). For import detail,
   `query_graph` with `MATCH (a:File)-[:IMPORTS]->(b:File) RETURN a.name,
   b.name` ‚Äî note IMPORTS edges connect File nodes, not Module nodes. If the
   project has more than ~25 modules, aggregate edges to cluster level (one
   diagram node per cluster).
4. Dead code: `search_graph` with `label: "Function"`, `max_degree: 0` (and
   again for `Method`), `limit: 30`. From the results DROP: symbols whose
   `file_path` is outside the repo (e.g. `<python-builtins>`), `is_test:
   true`, and `is_entry_point: true`. Flag `is_exported: true` rows as
   "public API ‚Äî verify before removal". Keep at most 30 rows.

Then write `ARCHITECTURE.md` with exactly this structure:

1. Marker line (first line, see below).
2. `# Architecture` title and a note: generated by `/architecture` and kept
   fresh automatically ‚Äî do not hand-edit, edits will be overwritten.
3. `## Overview` ‚Äî 2‚Äì3 paragraphs on what the system is and how it is shaped,
   derived from cluster/package names and the project README.
4. `## Languages & Packages` ‚Äî a table.
5. `## Module Map` ‚Äî one subsection per cluster: its purpose and key
   files/modules.
6. `## Key Entry Points` ‚Äî table: symbol, file, role.
7. `## HTTP Routes` ‚Äî table: method, path, handler symbol, file. Omit the
   section entirely if there are no routes.
8. `## Module Dependency Diagram` ‚Äî a mermaid `flowchart TD` block built from
   step 3 data (max ~25 nodes; collapse to clusters above that).
9. `## Dead Code Candidates` ‚Äî table, explicitly labelled
   "candidates (heuristic ‚Äî zero callers in the indexed graph)". Omit if none.

Finally, run `bash .claude/hooks/arch-doc-state.sh` (no arguments) and put its
output into the marker as the FIRST line of the file:

```
<!-- arch-doc commit=<sha> state=<hash> -->
```

## Incremental mode

1. Map the diff since the marker to affected symbols/files/modules:
   - if the server exposes a `detect_changes` tool, call it with
     `base = <marker commit>`;
   - otherwise (e.g. server v0.9.x): run
     `git diff --name-only <marker commit>` and, for each changed source
     file, `search_graph` with `file_pattern: "<path>"` to list its symbols
     and modules.
2. If the affected set is empty (only non-source churn), just refresh the
   marker line (run `bash .claude/hooks/arch-doc-state.sh` with no arguments)
   and report "no architecture-relevant changes".
3. If the affected set is large (more than ~30% of modules or ~50 files), fall
   back to full mode.
4. Otherwise update only what changed:
   - Module Map subsections for affected clusters;
   - HTTP Routes only if route-defining files changed;
   - the dead-code query re-run scoped to affected modules;
   - the mermaid diagram only if imports changed (module added/removed,
     import edges changed);
   - Overview and Languages & Packages refreshed from one `get_architecture`
     call (it is cheap).
5. Refresh the marker line: run `bash .claude/hooks/arch-doc-state.sh` (no
   arguments) and replace the first line of `ARCHITECTURE.md` with its output.

If the change-mapping step errors or behaves unexpectedly, fall back to full
mode.

## Rules

- Do not modify any files other than `ARCHITECTURE.md`.
- Keep the document factual and compact; no speculation about intent that the
  graph cannot support.
- Hard caps: 25 diagram nodes, 30 dead-code rows, 15 entry points.
